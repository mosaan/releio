「開発企画書」（Business Case）は、プロジェクトを進めることの経済的な正当性、つまり**プロジェクト遂行のための根拠**を確立し、提示する文書です。

この成果物は、開発作業の初期段階である**方向づけフェーズ**の主要な目標であり、推敲フェーズにかけて詳細化され、最終決定の根拠となります。

---

### 開発企画書（Business Case）に求められる内容の概要

開発企画書は、主に**開発提案（技術・コスト）**と**投資効果（経済的利益）**の二つの側面からプロジェクトを考察するものです。

#### 1. 開発提案/プロジェクトの計画と実現可能性（技術的側面）

開発企画書には、プロジェクトの実現可能性と、システムを構築するための計画の概要が含まれます。特に、初期の開発企画書は、その裏付けとして、以下のような情報に基づいています。

| 項目 | 求められる内容と目的 | 関連フェーズ/成果物 |
| :--- | :--- | :--- |
| **責任範囲の定義** | 提案するシステムに何が含まれ、何が含まれないかを正確に区別します。システムの境界と、外部のアクターやシステムとのインターフェイスの一般的な性質が記述されている必要があります。 | 方向づけフェーズ |
| **アーキテクチャ候補の概略** | システムの新規部分、リスクがある部分、難しい部分について記述されます。特に、次のフェーズで安定したアーキテクチャを作成できるという確信を得るための根拠（例えば、アーキテクチャの土台となる技術の選択が適切か、効率的か、信頼性が高いかなど）が含まれます。 | 方向づけフェーズ |
| **リスクの洗い出しと軽減計画** | プロジェクトの成功を妨げる**致命的リスク**を洗い出し、その優先順位を決定し、軽減策を略述します。リスクは、スケジュール、コスト、品質に与える影響に基づいて分類されます。 | 方向づけフェーズ |
| **初期の見積もり** | コスト、労力（リソースの種類や工数）、スケジュール、製品の品質についての大まかな初期見積もりが含まれます。当初は不確実なことが多いですが、推敲フェーズの終わりまでには高い精度で見積もれるように進展します。 | 方向づけフェーズ |
| **成功基準** | プロジェクトの販売予測や市場での認知度、プロジェクトの見積もりなどの**成功基準**が含まれます。 | 方向づけフェーズ |
| **ユースケースの概要** | 最も重要なユースケースの一部から成る単純なユースケースモデルが含まれ、システムが各主要ユーザーに対して主に行うことは何かという疑問に答えます。 | 方向づけフェーズ |
| **概念立証プロトタイプ** | 新しいシステムの基本的なアイデアやアルゴリズムを実証するために作成されたプロトタイプについての情報が含まれます。 | 方向づけフェーズ |

#### 2. 投資効果の評価（経済的側面）

開発企画書は、プロジェクトを進める経済的価値を証明するために、投資対効果（費用対効果）の評価を含みます。

*   **費用対効果の証明**: 開発企画書は、その開発コストを上回る収入や便益がシステムから得られるかどうかという経済的な観点を示します。
*   **市場投入時期**: 利益が得られる時期に間に合うように、ソフトウェア製品を市場に出荷できるか（または社内で適用できるか）というタイミングの要素も含まれます。
*   **利益の根拠**: 一般市場向け製品の場合、販売個数、販売単価などが検討材料となり、社内使用ソフトウェアの場合、対象部門の費用削減効果などが見積もられます。
*   **投資判断基準**: 上位の管理職がプロジェクトの続行を決定するための、投資効果のハードルや判断基準が示されることがあります。

#### フェーズごとの進捗

開発企画書は、ライフサイクルを通じて段階的に詳細化されます。

*   **方向づけフェーズ（Inception）の終わり**: **初期の開発企画書**が作成され、プロジェクトを推敲フェーズに進めることの正当性が示されます。この段階では、まだ大ざっぱな見積もりに基づいています。
*   **推敲フェーズ（Elaboration）の終わり**: **完成した開発企画書**が作成され、作成フェーズ以降に対する**開発提案書（固定価格の開発契約に相当）**を含む、信頼できる提案が準備されます。この段階では、コストとスケジュールが通常ビジネスで求められるより高い精度で提示されます。

### まとめ

開発企画書は、単なる技術的な仕様書ではなく、**プロジェクトがビジネスとして成立するかどうか**を判断するための総合的な文書です。これは、開発を組織化し、リスクを初期段階で管理するために、経済性、実現性、技術的な骨子（アーキテクチャ候補）をバランス良く示すことを要求されます。

例えるなら、開発企画書は、新事業の立ち上げにおける「事業計画書」と「技術的ロードマップ」を組み合わせたものです。事業の成功（投資効果）と、それを実現する実行可能な方法（アーキテクチャ、リスク対応、コスト）を同時に証明する役割を担います。
